# v1.3.0-beta.1 — ConnectionManager Refactor & Adaptive Backoff

**Type:** Beta — significant internal refactor targeting the recurring "device deafness" symptom.
**Backward compatible:** Yes — no breaking changes to entities, config entries, or automations.

## Why a Beta

The connection-layer changes in this release directly affect how the integration talks to the
device over TCP/telnet. The defaults (90 s socket reuse, single retry, TCP RST on error,
exponential backoff after consecutive failures) are tuned for the embedded Elios4you stack
based on analysis, but the only authoritative test is your real device over several days. Two
diagnostic sensors are **enabled by default** so you can confirm at a glance whether the
integration is healthy.

**Please watch these sensors for at least 48 hours before reporting back:**

| Sensor | What "healthy" looks like |
|---|---|
| **Connection State** | Mostly `ready`. **Should never be `backoff`** for sustained periods. |
| **Connection Consecutive Failures** | Stays at `0`. Any drift > 3 means the backoff is active. |
| **Connection Silent Timeouts** *(opt-in)* | Grows slowly. Rapid growth = device dropping commands. |
| **Connection Forced Aborts** *(opt-in)* | Equals the number of error-path closes. Growing = problems. |
| **Connection Reuse Hits** *(opt-in)* | Grows ~2 per poll cycle. Confirms the reuse window is doing its job. |

If you used to have to "kick" the device off WiFi 2–3 times a day, the goal of this release
is to make that go away. The diagnostic sensors are how we'll know whether it worked.

## What's Changed

### 🏗️ Architecture

- **New `connection_manager.py` module.** The connection lifecycle is now an explicit state
  machine — `DISCONNECTED → CONNECTING → READY → BACKOFF`, terminal `CLOSED` on unload — that
  owns the single TCP/telnet session, serialises every command, and decides when to reconnect,
  retry, abort, or back off.
- **`api.py` thinned to a parser/orchestrator.** Formats the `@dat` / `@sta` / `@inf` / `@rel`
  commands, parses responses, exposes the data dictionary that backs every sensor. No sockets,
  no retry logic — that all lives in the manager now. Public API (`async_get_data`,
  `telnet_set_relay`, `close`, `data`, `name`, `host`) is unchanged.
- Dropped the `check_port()` method and the unused `as_diagnostics()` helper.

### 🚀 Reliability (device-deafness mitigations)

- **Reuse window raised to 90 s** (was 25 s). The same TCP session now persists across the
  default 60 s poll cadence instead of opening a fresh socket every minute — drastically
  reducing pressure on the device's small socket table.
- **Retries dropped from 3 to 1** per command. Worst-case connect-storm on a flaky cycle is
  now ~6 SYNs instead of ~12.
- **TCP RST on error close** (`transport.abort()`) instead of FIN. When a command fails the
  socket is dropped with RST so the device frees its slot immediately rather than waiting out
  CLOSE_WAIT for its own internal timeout.
- **Bounded `wait_closed()`** (2 s). A misbehaving device can no longer hang the integration
  on close.
- **Exponential backoff** after 3 consecutive failures (5 s → 60 s, capped). While in backoff,
  the manager refuses to even attempt a new connection, giving the device room to recover.
  This is the single biggest change in behaviour.

### 📊 Diagnostics

- **12 new diagnostic sensors** under the device's Diagnostic section:
  - `Connection State` ✅ enabled by default
  - `Connection Consecutive Failures` ✅ enabled by default
  - `Connection Backoff Remaining`
  - `Connection Connects Succeeded`
  - `Connection Connect Failures`
  - `Connection Reuse Hits`
  - `Connection Commands Sent / Failed / Retried`
  - `Connection Silent Timeouts`
  - `Connection Forced Aborts`
  - `Connection Last Error`
- **Downloadable diagnostics** (Settings → Devices → ⋮ → Download diagnostics) now includes a
  `connection_manager` section with the full metrics snapshot.

### 🔧 Logging

- **Structured `(ConnMgr.*)` prefix** for every state transition, connect attempt, retry,
  close, and backoff event. To scope debug logging to just the manager:

  ```yaml
  logger:
    default: warning
    logs:
      custom_components.4noks_elios4you.connection_manager: debug
  ```

### 🧹 Code Quality

- Dropped every `# type: ignore` comment in the codebase. `DataUpdateCoordinator[bool]` and
  `CoordinatorEntity[Elios4YouCoordinator]` are now properly parameterised, and the Unicode
  telnetlib3 variants are `cast()` at the assignment site.
- **Test coverage**: 99 % overall, 100 % on `api.py`, `config_flow.py`, `switch.py`, and
  others. 211 tests pass.

### 📚 Documentation

- README rewritten: `### Technical Architecture` describes the new split; the troubleshooting
  log-format section uses real `(ConnMgr.*)` examples; the diagnostic sensors are documented.
- All pre-existing pymarkdown `MD032` violations in `CLAUDE.md`, `CHANGELOG.md`, `README.md`,
  and the full `docs/releases/` history are fixed — CI Lint is green again after months of
  red.
- `e4u-client/e4u.py` test client refreshed to mirror the new `ConnectionManager` patterns
  (RST on error, bounded `wait_closed`, structured constants).

### ⚠️ Breaking Changes

**None.** `api.py`'s public surface is unchanged. Existing config entries, sensors,
automations, and the relay switch keep working without modification. Migration from any prior
version is seamless.

## Upgrade & Rollback

**Upgrade**: HACS → Integrations → 4-noks Elios4you → Update to `1.3.0-beta.1`. Restart Home
Assistant. The two default-enabled diagnostic sensors will appear on the device page.

**Rollback**: if something is wrong, downgrade to `1.2.0` via HACS. Config entries are
forward-compatible — no migration loss.

---

## How to Help Validate

1. Note your `Connection Consecutive Failures` and `Connection Silent Timeouts` over 48 h.
2. Compare against your past baseline (how many WiFi-kicks per day before).
3. Open a GitHub issue if you see:
   - `Connection Consecutive Failures > 0` sustained
   - `Connection Silent Timeouts` climbing more than ~5/hour
   - The device going "deaf" at the same rate as before

A diagnostics download (Settings → Devices → 4-noks Elios4you → ⋮ → Download diagnostics)
attached to the issue includes everything we need.
